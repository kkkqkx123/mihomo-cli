# 需求规格说明：Mihomo CLI 管理工具

## 1. 概述

### 1.1 项目背景
本项目旨在开发一个非交互式的 Mihomo CLI 管理工具，用于 Windows 环境。该工具通过命令行界面提供对 Mihomo 代理核心的完整管理能力，包括运行模式切换、代理节点管理、Windows 服务控制等功能。

### 1.2 系统范围
**包含范围：**
- 命令行界面管理功能
- Mihomo RESTful API 交互
- Windows 服务管理
- 本地配置持久化

**不包含范围：**
- 图形用户界面 (GUI)
- 跨平台支持（仅限 Windows）

### 1.3 用户特征
- 系统管理员：需要批量管理多台机器的代理配置
- 高级用户：希望通过脚本自动化管理代理行为
- 开发人员：需要集成代理管理功能到自动化流程中

## 2. 功能需求

### 2.1 模式管理 (Mode Management)

#### REQ-MODE-001: 查询运行模式
**描述：** 用户能够查询当前 Mihomo 的运行模式。

**验收标准：**
- [EARS-1] 当用户执行 `mihomo-cli mode get` 命令时，系统应返回当前运行模式（rule/global/direct）。
- [EARS-2] 当 API 连接失败时，系统应返回错误信息并设置非零退出码。
- [EARS-3] 当用户指定 `-o json` 参数时，系统应以 JSON 格式输出结果。

#### REQ-MODE-002: 切换运行模式
**描述：** 用户能够切换 Mihomo 的运行模式。

**验收标准：**
- [EARS-4] 当用户执行 `mihomo-cli mode set <mode>` 命令时，系统应将运行模式切换为指定值（rule/global/direct）。
- [EARS-5] 当指定的模式无效时，系统应返回错误提示并列出有效选项。
- [EARS-6] 当切换成功时，系统应显示确认信息。

### 2.2 代理管理 (Proxy Management)

#### REQ-PROXY-001: 列出代理节点
**描述：** 用户能够列出所有可用的代理节点。

**验收标准：**
- [EARS-7] 当用户执行 `mihomo-cli proxy list` 命令时，系统应返回所有代理组的节点列表。
- [EARS-8] 当用户指定代理组名称时，系统应仅返回该代理组的节点列表。
- [EARS-9] 当用户指定 `-o json` 参数时，系统应以 JSON 格式输出结果。
- [EARS-10] 当输出格式为表格时，系统应显示节点名称、延迟、类型等关键信息。

#### REQ-PROXY-002: 切换代理节点
**描述：** 用户能够切换当前使用的代理节点。

**验收标准：**
- [EARS-11] 当用户执行 `mihomo-cli proxy switch <group> <node>` 命令时，系统应将指定代理组切换到指定节点。
- [EARS-12] 当代理组或节点不存在时，系统应返回错误信息。
- [EARS-13] 当切换成功时，系统应显示确认信息。

#### REQ-PROXY-003: 测试节点延迟
**描述：** 用户能够测试代理节点的延迟。

**验收标准：**
- [EARS-14] 当用户执行 `mihomo-cli proxy test <group> [node]` 命令时，系统应测试指定节点或代理组内所有节点的延迟。
- [EARS-15] 当测试完成时，系统应显示各节点的延迟结果。
- [EARS-16] 当节点不可达时，系统应标记延迟为超时状态。

#### REQ-PROXY-004: 自动选择最快节点
**描述：** 用户能够自动选择延迟最低的节点。

**验收标准：**
- [EARS-17] 当用户执行 `mihomo-cli proxy auto <group>` 命令时，系统应测试代理组内所有节点延迟并自动切换到最快节点。
- [EARS-18] 当所有节点均不可达时，系统应返回警告信息并保持当前节点不变。

#### REQ-PROXY-005: 取消固定代理
**描述：** 用户能够取消代理组中固定的代理，恢复自动选择模式。

**验收标准：**
- [EARS-121] 当用户执行 `mihomo-cli proxy unfix <group>` 命令时，系统应取消指定代理组的固定选择，恢复自动选择。
- [EARS-122] 当代理组不存在时，系统应返回错误提示。
- [EARS-123] 当取消成功时，系统应显示确认信息。

**验收标准：**

- [EARS-60] 当配置项为 tun 或 tuic-server 时，系统应重新创建相应监听器。
- [EARS-61] 当更新成功时，系统应显示确认信息。
- [EARS-62] 当 API 不支持该配置项热更新时，系统应返回错误提示。

#### REQ-CFG-005: 完整配置重载
**描述：** 用户能够重新加载完整的 Mihomo 配置文件。

**验收标准：**
- [EARS-63] 当用户执行 `mihomo-cli config reload` 命令时，系统应重新加载当前配置文件。
- [EARS-64] 当用户指定 `--path <path>` 参数时，系统应从指定路径加载配置文件。
- [EARS-65] 当用户指定 `--force` 参数时，系统应强制重载配置，忽略部分错误。
- [EARS-66] 当配置文件路径不是绝对路径时，系统应返回错误提示。
- [EARS-67] 当配置文件路径不在安全路径范围内时，系统应返回安全错误提示。
- [EARS-68] 当重载成功时，系统应显示确认信息。

#### REQ-CFG-006: 配置文件编辑
**描述：** 用户能够编辑 Mihomo 配置文件并自动重载。

**验收标准：**
- [EARS-69] 当用户执行 `mihomo-cli config edit <key> <value>` 命令时，系统应读取现有配置文件、合并新配置值、备份原文件并写入新配置。
- [EARS-70] 当配置文件不存在时，系统应返回错误提示。
- [EARS-71] 当备份文件创建失败时，系统应中止操作并返回错误信息。
- [EARS-72] 当配置写入成功后，系统应自动调用重载命令使配置生效。
- [EARS-73] 当用户指定 `--no-reload` 参数时，系统应仅修改文件不触发重载。
- [EARS-74] 当 YAML 格式错误时，系统应返回解析错误信息。

#### REQ-CFG-007: 批量配置更新
**描述：** 用户能够通过 JSON 或 YAML 输入批量更新多个配置项。

**验收标准：**
- [EARS-75] 当用户执行 `mihomo-cli config patch --file <path>` 命令时，系统应从指定文件读取配置并批量更新。
- [EARS-76] 当用户通过标准输入传入配置时，系统应读取并批量更新。
- [EARS-77] 当批量更新中部分配置项失败时，系统应显示成功和失败的配置项列表。

- [EARS-53] 当用户执行 `mihomo-cli <command> --help` 命令时，系统应显示该命令的详细用法。

## 5. 术语表

| 术语 | 定义 |
|------|------|
| Mihomo | 一个基于 Clash 的代理核心 |
| Rule 模式 | 规则模式，根据规则文件决定流量走向 |
| Global 模式 | 全局代理模式，所有流量通过代理 |
| Direct 模式 | 直连模式，所有流量不经过代理 |
| 代理组 | 一组代理节点的集合，可从中选择一个节点使用 |
| 订阅 | 从远程 URL 获取代理配置的方式 |
| 热更新 | 运行时修改配置，无需重启服务即可生效 |
| 配置重载 | 重新加载完整配置文件，可能影响运行状态 |
| 安全路径 | Mihomo 允许访问的配置文件路径范围，防止路径遍历攻击 |
