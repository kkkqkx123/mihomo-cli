# Mihomo CLI 不可恢复问题分析

## 概述

本文档基于 `mihomo-1.19.21\docs\system-proxy-modifications.md` 的分析，识别当前CLI项目中可能出现不可恢复问题的操作场景，特别是那些可能导致系统配置无法正常恢复或进程未能正常退出的情况。

## 术语定义

- **不可恢复问题**: 操作失败后，系统配置（如注册表、路由表、iptables规则、虚拟网卡等）未能恢复到正常状态，需要手动干预
- **正常退出**: 进程或操作完成后，所有系统配置已正确清理，没有残留的配置项

## 分析范围

本文档分析以下操作场景：
1. 系统代理管理（sysproxy命令）
2. Mihomo内核进程管理（start/stop/status命令）
3. Mihomo配置管理（mihomo命令）
4. 配置文件编辑（mihomo edit命令）

---

## 1. 系统代理管理操作

### 1.1 `sysproxy set on` - 启用系统代理

**操作描述**: 修改Windows注册表，启用系统代理设置

**系统配置修改**:
- 注册表路径: `HKEY_CURRENT_USER\SOFTWARE\Microsoft\Windows\CurrentVersion\Internet Settings`
- 修改的注册表项:
  - `ProxyEnable` = 1 (DWORD)
  - `ProxyServer` = "127.0.0.1:7890" (SZ)
  - `ProxyOverride` = "localhost;127.*;10.*;172.16.*;172.31.*;192.168.*" (SZ)

**潜在不可恢复场景**:

#### 场景1.1.1: 进程被强制中断
- **触发条件**: 用户按 Ctrl+C 终止命令，或进程被系统强制终止
- **问题描述**:
  - 注册表已修改为启用状态
  - 但代理服务器可能不可用（127.0.0.1:7890未监听）
  - 系统所有应用（IE/Edge/使用WinHTTP的应用）无法访问网络
- **恢复方法**: 手动运行 `sysproxy set off` 或通过Windows设置禁用系统代理
- **严重程度**: 高（影响所有网络连接）

#### 场景1.1.2: 权限不足导致部分配置修改失败
- **触发条件**: 用户权限不足，注册表部分写入失败
- **问题描述**:
  - `ProxyEnable` 写入成功，但 `ProxyServer` 或 `ProxyOverride` 写入失败
  - 系统代理已启用，但配置不完整
  - 可能导致网络请求异常或无法正确连接代理
- **恢复方法**: 手动运行 `sysproxy set off` 清理
- **严重程度**: 中

#### 场景1.1.3: 配置参数错误
- **触发条件**: 提供的 `--server` 或 `--bypass` 参数格式错误
- **问题描述**:
  - 注册表写入成功，但代理服务器地址无效
  - 所有网络请求失败
- **恢复方法**: 手动运行 `sysproxy set off` 或修复代理服务器地址
- **严重程度**: 高

### 1.2 `sysproxy set off` - 禁用系统代理

**操作描述**: 恢复Windows注册表，禁用系统代理设置

**系统配置恢复**:
- 设置 `ProxyEnable` = 0
- 可选：清除 `ProxyServer` 和 `ProxyOverride`

**潜在不可恢复场景**:

#### 场景1.2.1: 注册表操作失败
- **触发条件**: 注册表权限不足或被其他进程锁定
- **问题描述**:
  - 无法禁用系统代理
  - 即使Mihomo进程已停止，系统代理仍然启用
  - 可能导致网络请求失败
- **恢复方法**:
  1. 检查注册表权限
  2. 关闭可能锁定注册表的进程
  3. 通过Windows设置手动禁用代理
- **严重程度**: 高

## 2. Mihomo内核进程管理操作

### 2.1 `start` - 启动Mihomo内核

**操作描述**: 启动Mihomo内核进程，生成随机密钥，进行健康检查

**系统配置影响**:
如果Mihomo配置文件启用了以下功能，则会影响系统配置：
- **标准系统代理**: 修改注册表（与CLI的sysproxy命令独立）
- **TUN模式**: 创建虚拟网卡，修改路由表
- **TProxy模式**: 修改iptables规则（仅Linux）

**潜在不可恢复场景**:

#### 场景2.1.1: 启动成功但健康检查超时
- **触发条件**: Mihomo进程启动成功，但API无法响应（配置错误、端口冲突等）
- **代码位置**: `internal/mihomo/process_handler.go:Start()`
- **问题描述**:
  ```go
  // 健康检查超时处理
  case <-checkCtx.Done():
      // 健康检查超时，通过 PID 停止进程
      if pid, err := pm.GetPIDFromPIDFile(); err == nil {
          _ = StopProcessByPID(pid)
      }
      return nil, pkgerrors.ErrService("mihomo health check timeout: process may have failed to start", nil)
  ```
  - 进程已启动
  - CLI尝试通过PID停止进程
  - **但如果Mihomo配置启用了TUN/TProxy模式**，以下清理操作可能不完整：
    - TUN虚拟网卡未删除
    - 路由表未恢复
    - iptables规则未清理
  - 导致系统网络配置残留
- **恢复方法**:
  1. 手动删除TUN虚拟网卡
  2. 手动清理路由表
  3. 手动清理iptables规则
  4. 或重启系统
- **严重程度**: 极高（影响系统网络）

#### 场景2.1.2: 启动后立即收到中断信号
- **触发条件**: 进程启动后，用户按Ctrl+C或系统发送SIGTERM
- **代码位置**: `cmd/start.go:runStart()`
- **问题描述**:
  ```go
  case sig := <-sigChan:
      // 收到中断信号
      fmt.Printf("\n收到信号 %v，正在停止内核...\n", sig)

      // 通过 PID 文件停止进程
      pm := mihomo.NewProcessManager(cfg)
      if pid, err := pm.GetPIDFromPIDFile(); err == nil {
          if err := mihomo.StopProcessByPID(pid); err != nil {
              fmt.Printf("停止内核失败: %v\n", err)
              return err
          }
      }
  ```
  - 进程被强制终止（SIGKILL）
  - **Mihomo内核没有机会执行清理操作**：
    - TUN虚拟网卡无法删除
    - 路由表无法恢复
    - iptables规则无法清理
    - 注册表代理设置无法禁用（如果Mihomo启用了系统代理）
- **恢复方法**: 同场景2.1.1
- **严重程度**: 极高

#### 场景2.1.3: 进程异常退出（崩溃）
- **触发条件**: Mihomo进程因配置错误或内部bug崩溃
- **代码位置**: `internal/mihomo/manager.go:Start()`
- **问题描述**:
  ```go
  go func() {
      err := pm.cmd.Wait()
      pm.mu.Lock()
      pm.isRunning = false
      // 进程退出时删除 PID 文件
      os.Remove(pm.pidFile)
      
      // 如果进程异常退出，输出错误信息
      if err != nil {
          fmt.Printf("\n[Mihomo 进程异常退出] 错误: %v\n", err)
          // ... 输出错误信息
      }
      pm.mu.Unlock()
  }()
  ```
  - 进程已退出
  - PID文件被删除
  - **但Mihomo内核的系统配置未被清理**：
    - TUN虚拟网卡残留
    - 路由表残留
    - iptables规则残留
    - 注册表代理设置残留
- **恢复方法**: 同场景2.1.1
- **严重程度**: 极高

#### 场景2.1.4: 配置文件中的TUN/TProxy配置启动后未正常退出
- **触发条件**: 配置文件中启用了 `tun.enable: true` 或启用TProxy模式
- **问题描述**:
  - TUN虚拟网卡已创建
  - 路由表已修改（添加自定义路由表）
  - 如果进程异常退出或被强制终止
  - 网卡和路由表配置残留
  - 可能导致：
    - 无法连接到某些网络（路由被劫持到虚拟网卡）
    - DNS解析失败（如果启用了DNS重定向）
    - 系统网络异常
- **恢复方法**:
  1. Windows: 手动删除TUN网卡（网络适配器设置）
  2. Linux: 手动删除TUN网卡和路由规则
  3. 重启系统（最简单可靠）
- **严重程度**: 极高

### 2.2 `stop` - 停止Mihomo内核

**操作描述**: 停止Mihomo内核进程

**潜在不可恢复场景**:

#### 场景2.2.1: 进程停止失败
- **触发条件**: 进程无响应或权限不足
- **代码位置**: `internal/mihomo/manager.go:StopProcessByPID()`
- **问题描述**:
  ```go
  if err := proc.Kill(); err != nil {
      return pkgerrors.ErrService("failed to stop process", err)
  }
  ```
  - Kill命令执行失败
  - 进程仍在运行
  - PID文件未被删除
  - 系统配置未被清理
- **恢复方法**:
  1. 使用任务管理器强制结束进程
  2. 手动清理系统配置（注册表、路由表、iptables）
  3. 重启系统
- **严重程度**: 高

#### 场景2.2.2: 停止后PID文件删除失败
- **触发条件**: PID文件被锁定或权限不足
- **代码位置**: `internal/mihomo/process_handler.go:Stop()`
- **问题描述**:
  ```go
  // 删除 PID 文件
  os.Remove(pm.pidFile)
  ```
  - 进程已停止
  - PID文件未删除
  - 下次启动时可能误认为进程仍在运行
  - 导致无法启动新进程
- **恢复方法**: 手动删除PID文件
- **严重程度**: 中

#### 场景2.2.3: `--all` 参数停止所有进程时的清理不完整
- **触发条件**: 多个Mihomo进程同时运行，使用 `--all` 停止
- **问题描述**:
  - 停止所有Mihomo进程
  - 如果某个进程的系统配置未被清理（TUN网卡、路由表等）
  - 导致配置残留
- **恢复方法**: 手动清理所有残留配置
- **严重程度**: 高

## 3. Mihomo配置管理操作

### 3.1 `mihomo reload` - 重载配置文件

**操作描述**: 通过API重载Mihomo配置文件

**系统配置影响**:
- 如果新配置启用了TUN/TProxy模式，会创建新的网络配置
- 如果旧配置有TUN/TProxy，新配置没有，应该清理旧配置

**潜在不可恢复场景**:

#### 场景3.1.1: 重载过程中进程崩溃
- **触发条件**: 配置文件有语法错误或逻辑错误
- **问题描述**:
  - 旧配置的部分系统配置已清理
  - 新配置的系统配置未应用
  - 进程崩溃
  - 导致系统配置处于不一致状态
- **恢复方法**: 重启Mihomo进程
- **严重程度**: 高

#### 场景3.1.2: 重载后网络配置切换失败
- **触发条件**: 从TUN模式切换到标准代理模式，或反之
- **问题描述**:
  - 旧的TUN网卡未删除
  - 新的系统代理设置未生效
  - 导致网络配置混乱
- **恢复方法**: 手动清理旧配置，重启进程
- **严重程度**: 高

### 3.2 `mihomo patch` - 热更新配置

**操作描述**: 通过API热更新Mihomo运行时配置

**潜在不可恢复场景**:

#### 场景3.2.1: 热更新TUN配置后进程崩溃
- **触发条件**: 修改TUN相关配置项
- **问题描述**:
  - TUN网络配置修改到一半
  - 进程崩溃
  - 导致网络配置不一致
- **恢复方法**: 重启Mihomo进程
- **严重程度**: 高

## 4. 配置文件编辑操作

### 4.1 `mihomo edit` - 编辑配置文件

**操作描述**: 编辑Mihomo配置文件并自动重载

**潜在不可恢复场景**:

#### 场景4.1.1: 编辑后重载失败
- **触发条件**: 配置文件编辑后，重载API调用失败
- **代码位置**: `cmd/mihomo.go:runMihomoEdit()`
- **问题描述**:
  ```go
  // 重载配置
  if err := client.ReloadConfig(ctx, configPath, false); err != nil {
      color.Yellow("警告: 重载配置失败: %v", err)
      color.Yellow("配置文件已修改，但未生效，请手动重启服务")
      return nil
  }
  ```
  - 配置文件已修改
  - 但重载失败
  - 如果用户修改了TUN/TProxy配置，进程重启后可能应用新配置
  - 但如果进程异常退出，新配置可能未被清理
- **恢复方法**: 手动重启Mihomo进程
- **严重程度**: 中

#### 场景4.1.2: 编辑后使用 `--no-reload` 参数
- **触发条件**: 用户编辑配置文件但不触发重载
- **问题描述**:
  - 配置文件已修改
  - 但Mihomo仍在使用旧配置
  - 用户手动重启Mihomo时，如果配置有错误
  - 导致进程启动失败或配置残留
- **恢复方法**: 检查配置文件，修复错误后重启
- **严重程度**: 中

## 5. 跨操作复合场景

### 5.1 CLI系统代理 + Mihomo系统代理冲突

**场景描述**:
1. 用户运行 `sysproxy set on` 启用CLI系统代理
2. 用户运行 `mihomo edit` 启用Mihomo的系统代理功能
3. Mihomo进程崩溃或被强制终止
4. 两个系统代理设置都残留

**问题描述**:
- CLI启用的系统代理设置在注册表中
- Mihomo启用的系统代理设置也在注册表中（可能覆盖CLI的设置）
- 进程退出后，注册表中的代理设置未恢复
- 导致网络连接异常

**恢复方法**:
1. 运行 `sysproxy set off` 清理
2. 或通过Windows设置手动禁用代理

**严重程度**: 高

### 5.2 TUN模式 + 系统代理同时启用

**场景描述**:
1. Mihomo配置启用了TUN模式
2. 用户运行 `sysproxy set on` 启用系统代理
3. Mihomo进程崩溃或被强制终止

**问题描述**:
- TUN虚拟网卡残留
- 路由表残留
- 注册表系统代理设置残留
- 网络配置完全混乱
- 可能无法正常上网

**恢复方法**:
1. 手动删除TUN网卡
2. 手动清理路由表
3. 运行 `sysproxy set off`
4. 或重启系统

**严重程度**: 极高

### 5.3 快速连续操作导致状态不一致

**场景描述**:
1. 用户运行 `start` 启动Mihomo
2. 立即按Ctrl+C停止
3. 立即再次运行 `start`
4. 再次按Ctrl+C停止

**问题描述**:
- PID文件可能处于不一致状态
- 进程可能未完全退出
- 系统配置可能部分清理
- 导致下次启动失败或配置残留

**恢复方法**:
1. 手动检查并清理所有Mihomo进程
2. 删除所有PID文件
3. 清理系统配置（注册表、路由表、iptables）
4. 重新启动

**严重程度**: 高

## 6. 根本原因分析

### 6.1 缺少优雅关闭机制

**问题描述**:
- CLI使用 `proc.Kill()` 强制终止进程
- 没有给Mihomo内核执行清理操作的机会
- 导致TUN网卡、路由表、iptables规则、注册表设置等无法清理

**代码证据**:
```go
// internal/mihomo/manager.go:StopProcessByPID()
if err := proc.Kill(); err != nil {
    return pkgerrors.ErrService("failed to stop process", err)
}
```

**建议改进**:
1. 优先尝试通过API发送优雅关闭请求
2. 设置超时时间，如果超时再使用Kill
3. 或使用SIGTERM信号，等待进程优雅退出

### 6.2 信号处理不完整

**问题描述**:
- `start` 命令捕获了SIGINT和SIGTERM信号
- 但处理逻辑中直接调用 `StopProcessByPID()`，也是强制终止
- 没有给Mihomo内核执行清理的机会

**代码证据**:
```go
// cmd/start.go:runStart()
case sig := <-sigChan:
    // 收到中断信号
    fmt.Printf("\n收到信号 %v，正在停止内核...\n", sig)
    
    // 通过 PID 文件停止进程
    pm := mihomo.NewProcessManager(cfg)
    if pid, err := pm.GetPIDFromPIDFile(); err == nil {
        if err := mihomo.StopProcessByPID(pid); err != nil {
            fmt.Printf("停止内核失败: %v\n", err)
            return err
        }
    }
```

**建议改进**:
1. 捕获信号后，先通过API发送优雅关闭请求
2. 等待进程自然退出
3. 如果超时，再使用Kill强制终止

### 6.3 缺少系统配置状态检查

**问题描述**:
- CLI没有检查Mihomo配置中是否启用了TUN/TProxy模式
- 没有在启动前提醒用户可能的风险
- 没有在停止后验证系统配置是否已清理

**建议改进**:
1. 在启动前检查配置文件，如果启用了TUN/TProxy，发出警告
2. 在停止后检查系统配置状态，如果未清理，提示用户手动清理
3. 提供清理命令（如 `mihomo-cli cleanup`）

### 6.4 缺少配置验证

**问题描述**:
- `mihomo edit` 和 `mihomo reload` 没有充分验证配置文件
- 如果配置文件有错误，可能导致进程崩溃或配置残留

**建议改进**:
1. 在重载前先验证配置文件语法
2. 如果启用TUN/TProxy，验证相关配置项
3. 提供配置测试命令

### 6.5 错误处理不完善

**问题描述**:
- 部分操作失败后，错误信息不够详细
- 用户不知道如何恢复
- 没有提供自动恢复建议

**建议改进**:
1. 所有错误信息都应包含恢复建议
2. 提供自动恢复命令
3. 记录详细的操作日志

## 7. 风险评估总结

### 7.1 按严重程度分类

| 严重程度 | 场景数量 | 主要影响 |
|---------|---------|---------|
| 极高 | 4 | 系统网络配置残留，需要重启系统才能恢复 |
| 高 | 8 | 部分系统配置残留，需要手动清理 |
| 中 | 4 | 影响功能使用，但不影响系统网络 |

### 7.2 按操作类型分类

| 操作类型 | 风险场景数 | 最高严重程度 |
|---------|-----------|-------------|
| 系统代理管理 | 3 | 高 |
| Mihomo进程管理 | 6 | 极高 |
| 配置管理 | 3 | 高 |
| 配置编辑 | 2 | 中 |

### 7.3 按配置模式分类

| 配置模式 | 风险场景数 | 主要风险 |
|---------|-----------|---------|
| 标准系统代理 | 5 | 注册表设置残留 |
| TUN模式 | 4 | 虚拟网卡和路由表残留 |
| TProxy模式 | 2 | iptables规则残留 |

## 8. 建议的改进措施

### 8.1 短期改进（立即实施）

1. **改进停止机制**
   - 优先通过API发送优雅关闭请求
   - 设置超时时间（如10秒）
   - 超时后再使用Kill强制终止

## 10. 结论

当前CLI项目在以下操作中可能出现不可恢复问题：

1. **Mihomo内核启动和停止**：特别是启用了TUN/TProxy模式时
2. **系统代理管理**：进程被中断时可能导致注册表设置残留
3. **配置重载**：重载过程中进程崩溃可能导致配置不一致
4. **快速连续操作**：可能导致状态不一致和配置残留

主要原因是：
- 缺少优雅关闭机制
- 信号处理不完整
- 缺少系统配置状态检查
- 缺少配置验证
- 错误处理不完善

建议优先实施短期改进措施，逐步完善系统的健壮性和可恢复性。
